# 기술적으로 가장 어려웠던 문제와 극복 과정

## 문제 상황
콘서트 좌석 예약 시 사용자가 여러 좌석(예: 1, 2, 3번)을 동시에 예약하는 경우, 각 좌석별로 분산락이 필요했습니다. 만약 스케줄 전체를 락으로 잡으면 각각 다른 좌석을 예약하는 요청도 병렬 처리가 불가능해 성능이 저하되고, 좌석별로 개별 락을 사용할 경우 데드락이 발생할 수 있었습니다. 예를 들어 사용자 A가 [3, 1, 2]번 순서로, 사용자 B가 [1, 2, 3]번 순서로 락을 획득하면 서로 대기하는 데드락이 발생합니다. 분산환경을 고려했기 때문에 synchronized 같은 JVM 락은 적용할 수 없었습니다. 처음에는 RDB 임시 예약 테이블로 좌석 선점을 관리하려 했으나, 트랜잭션 충돌과 롤백 처리가 복잡했습니다. 이에 Redisson 분산락을 도입했지만, Adapter 레벨에서 수동으로 여러 락을 관리하는 120줄의 복잡한 코드가 생겼고, 트랜잭션보다 락이 늦게 실행되어 DB 커넥션을 낭비했습니다.

## 해결 과정
AOP 기반 다중 분산락 시스템으로 전면 리팩토링했습니다. @DistributedLock 어노테이션에 SpEL 표현식으로 좌석 ID 리스트를 전달하면, AOP가 각 좌석별 락 키(lock:seat:1:1, lock:seat:1:2)를 자동 생성합니다. 데드락 방지를 위해 sorted=true 옵션으로 모든 스레드가 [1, 2, 3] 순서로 정렬된 락을 획득하도록 했습니다. @Order 설정으로 분산락을 트랜잭션보다 먼저 실행하여 DB 커넥션 효율을 개선했습니다.

## 결과
통합 테스트에서 100명이 동일 좌석 동시 예약 시 1명만 성공하고, 서로 다른 좌석은 병렬 처리되는 것을 검증했습니다. DB 커넥션 점유 시간이 75% 감소했고, 코드는 120줄에서 48줄로 60% 단순화되었습니다. 좌석별 세밀한 동시성 제어로 처리량과 안정성을 모두 확보했습니다.
