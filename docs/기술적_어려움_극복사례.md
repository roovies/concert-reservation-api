# 기술적으로 가장 어려웠던 문제와 극복 과정

## 문제 상황
콘서트 예약 시스템에서 동일한 결제 요청이 100번 동시에 발생할 때 중복 결제를 방지하는 멱등성 보장이 가장 큰 과제였습니다. 네트워크 재시도나 사용자의 중복 클릭으로 인해 같은 요청이 여러 번 전송될 수 있는데, 이를 제어하지 않으면 동일 좌석에 대해 여러 건의 결제가 발생할 수 있었습니다. Redis 분산락으로도 해결 가능하지만, 결제라는 중요한 트랜잭션에서는 DB와 락의 일관성 보장이 더 중요했습니다.

## 해결 과정
멱등성 키(idempotency key)를 DB 테이블의 Primary Key로 설정하여 Unique 제약조건을 활용했습니다. 클라이언트가 전송한 고유 키를 DB에 INSERT 시도하면, 첫 번째 요청만 성공하고 나머지는 DataIntegrityViolationException이 발생합니다. 이를 catch하여 중복 요청임을 감지하고, 처리 상태(PROCESSING, SUCCESS, FAILED)에 따라 적절히 응답합니다. 성공한 요청의 결과는 JSON으로 직렬화하여 저장하고, 중복 요청 시 동일한 결과를 반환하여 멱등성을 보장했습니다.

## 결과
통합 테스트에서 100개 스레드가 동일한 멱등성 키로 동시 결제 시 1건만 성공하고 99건은 "중복 요청" 예외 처리를 검증했습니다. DB 트랜잭션과 완벽히 통합되어 일관성이 보장되며, Redis 같은 외부 의존성 없이 간단하게 구현할 수 있었습니다. 결제 안정성과 사용자 경험이 크게 향상되었습니다.
